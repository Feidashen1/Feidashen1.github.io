# 1.内联函数
C++ 内联函数的主要作用是通过在编译阶段将函数代码直接替换到调用处来消除函数调用开销，从而提高程序的运行效率。它旨在用空间换时间，避免常规函数调用带来的性能损耗。

当编译器内联展开一个函数调用时，该函数的代码会被插入到调用点的代码流中（概念上类似于展开 `#define` 宏）。这通常能改善性能（但也受其它因素影响），因为优化器可以把被调用的代码更好地与调用代码一起优化。


## 1.1 内联函数定义

**显式内联**：

```cpp
inline 返回类型 函数名(参数列表) {
    // 函数体
}
```

* 内联机制最适合函数体简短、执行频繁的函数。如果函数很复杂或庞大，进行内联展开反而会让可执行文件体积显著增大（代码膨胀），降低指令缓存的命中率，得不偿失。
* 跟普通函数不同，内联函数的完整定义（包括 inline 关键字）必须在每一个调用它的源文件都是可见的。所以，内联函数要定义在头文件（.h 或 .hpp）中

**隐式内联**：
- 短小且频繁调用的函数：如存取函数（getter/setter）、简单的数学运算等。
- 类内部定义的成员函数：直接在类的定义体中实现的成员函数，编译器会自动将其视为内联函数。
- `constexpr` / `consteval` 函数：带有 `constexpr`（C++11）或 `consteval`（C++20）修饰的函数隐式具备 `inline` 属性。

## 1.2 内联的编译器控制

**将函数标记为 inline 仅是向编译器发出的内联展开请求，编译器拥有最终的决策权，完全可以选择忽略该建议**；

> 编译器的主要目标是生成高效且体积合理的机器代码。如果内联某个函数会导致代码膨胀，反而会降低缓存命中率，从而拖慢运行速度。因此，编译器会根据具体情况权衡。

编译器忽略内联的常见原因：
1. 包含复杂控制流： 例如包含大量的 switch、while 或 for 循环。递归函数；
2. 编译器无法在编译期确定递归的具体深度，通常不进行内联；
3. 体积过大： 如果函数体本身代码量很大，内联会导致目标文件（或可执行文件）体积剧增。
4. 函数指针调用：如果通过函数指针调用内联函数，编译器无法在编译期确定具体地址，因此只能生成常规的函数调用。

示例：内联函数 vs 宏的多次求值问题：

```cpp
#define SQR_MACRO(x) ((x)*(x))
inline int sqr_func(int x) { return x * x; }

int a = 3;
int b1 = SQR_MACRO(++a); // ++a 会被求值两次，结果不符合预期
int a2 = 3;
int b2 = sqr_func(++a2); // ++a2 只求值一次，行为可预期
```

### 1.2.1 编译器内联限制对策：
绕过编译器的内联限制：

|方法|描述|使用场景|
| - | - | - |
|强制内联属性|在 GCC/Clang 中使用 __attribute__((always_inline))，或在 MSVC 中使用 __forceinline。|必须严格保证性能的关键路径（但滥用会导致代码体积暴增）。|
|宏定义|彻底绕过编译器机制，直接在预处理阶段进行文本替换。|仅适用于极简的逻辑封装，缺乏类型安全。|
|编译期求值|在 C++11 及更高版本中使用 constexpr，配合模板，强制在编译阶段计算结果。|针对常量表达式和类型推导的高级优化。|

禁止内联：除了请求内联，有时需要确保函数绝对不被内联（如用于调试、减少代码体积，或暴露函数地址供其他模块使用）：
- GCC / Clang: 使用 __attribute__((noinline))。
- MSVC: 使用 __declspec(noinline)

## 1.3 内联函数与宏函数的区别
| 特性 | #define 宏函数 | inline 内联函数 |
| :--- | :--- | :--- |
| **处理阶段** | **预处理阶段**（纯文本替换，编译器还没运行） | **编译阶段**（编译器负责替换和优化） |
| **类型检查** | **无**。不检查参数类型，容易隐式引发类型错误 | **有**。严格检查参数和返回值类型，类型安全 |
| **参数求值** | **多次求值**。每在宏体内出现一次就求值一次 | **一次求值**。求值后作为参数传入，符合函数常规 |
| **作用域** | **无作用域**。全局有效，可能污染命名空间，无法作为类成员 | **服从作用域**。可作为类的私有/公有成员函数 |
| **调试支持** | **极难调试**。源码中无法设置断点，报错信息晦涩 | **支持调试**。多数编译器支持对内联函数进行断点和单步调试 |



### 1.3.1 宏函数相比内联函数更适合的 5 个场景

**1.文本与符号操作**：利用 `#` 进行字符串化（将参数名转为字符串），或利用 `##` 进行符号连接（动态拼接生成新的变量名或函数名），这是内联函数完全无法做到的。
> 字符串化（#）：将传入的参数名转换为字符串字面量

**2.获取调用处的编译元数据**：在宏中使用 `__FILE__`、`__LINE__`、`__FUNCTION__` 等内置宏，可以精准获取**调用发生处**的文件名和行号，常用于编写断言（Assert）和日志系统；而内联函数只会返回其自身定义处的信息。\
**3.彻底的条件编译与平台隔离**：宏函数配合 `#ifdef` 可以根据不同平台（如 Windows/Linux）注入完全不同的底层 API 或语法结构。在不满足条件的平台上，宏代码在预处理阶段就会被彻底抹除，避免引发编译错误。\
**4.C 语言绝对兼容与混编**：在纯 C 语言项目（尤其是老旧的 C89 标准底层固件）中，或者在需要同时被 C 和 C++ 编译器无缝调用的底层头文件中，宏函数具备比 `inline` 更好的跨语言兼容性。\
**5.复杂的代码模板生成**：在需要批量生成结构相似的结构体、枚举映射表或注册函数时（类似于初级的元编程或反射模拟），利用宏函数进行大段代码的展开和自动化生成更为高效。


> 决策法则：\
> 如果需要做类型转换、数学计算、逻辑判断：请一律使用 inline / template / constexpr。\
> 如果需要做代码生成、日志打桩、反射模拟、平台隔离：请果断使用宏函数



# 2.重载函数
函数重载（Function Overloading）是 C++ 允许在同一作用域内声明多个同名函数的机制。这些同名函数的参数列表（参数个数、类型或顺序）必须不同

编译器会根据调用时传入的实参自动匹配并调用最合适的函数版本，以此实现“一名前端，多重实现”的多态特性。

**返回值类型不能作为重载的依据**

> 底层原理：名字修饰（Name Mangling）\
为什么同名函数不会导致命名冲突？因为 C++ 编译器在编译时对函数名进行了名字修饰\
编译器会把函数的参数类型信息秘密地拼接到函数名中，生成全局唯一的内部名字，在链接阶段，它们本质上已经是两个完全不同的函数了。

示例：返回类型不能作为重载依据

```cpp
int f(int);
double f(int); // 错误：仅返回类型不同不能构成重载
```

重载陷阱提示：当多个重载通过不同的标准转换都可行时，可能出现二义性；在可疑处使用显式类型或 `static_cast`。

# 3.默认参数
函数的默认参数（Default Arguments）是 C++ 允许在函数声明或定义时，为某些形参指定一个缺省值。如果调用函数时没有提供对应的实参，编译器就会自动使用这个默认值

| 特性维度 | 规则与原理说明 | 错误示例 / 避坑提示 |
| :--- | :--- | :--- |
| **底层本质** | 纯编译期语法糖，编译器在调用点自动补全缺省实参，汇编层面压栈开销不变。 | 无运行时性能优势（空间和时间开销与普通调用一致）。 |
| **定义顺序** | 必须**从右向左**连续指定，右侧参数没有默认值时，左侧绝不能先有。 | `void func(int a = 1, int b);` ❌ (编译报错) |
| **指定位置** | 声明（头文件）和定义（源文件）中**只能指定一次**，推荐写在声明中。 | 两处都写：`void f(int a=1); void f(int a=1){}` ❌ (重复定义) |
| **虚函数动态绑定** | 默认参数是**静态绑定**的。通过基类指针调用派生类虚函数时，使用的是**基类**的默认参数！ | 绝不要在虚函数中重定义不同的默认参数，会导致逻辑混乱。 



## 3.1 默认参数两条底层铁律的成因

| 语法规则 | 对应的编译器底层原理 | 违反后果的本质 |
| :--- | :--- | :--- |
| **自右向左连续指定** | **基于位置的单向参数压栈算法**。C++ 缺乏命名参数机制，编译器必须通过绝对位置来一一对应地补全尾部参数。 | 若不连续，词法分析器（Parser）在遇到隐式转换或同类型参数时，会因无法推导产生词法二义性。 |
| **声明与定义不重复** | **独立编译单元的一致性判定**。防止多个 `.cpp` 文件在相互隔离的编译期阶段，由于看到不同的缺省值而生成逻辑冲突的汇编指令。 | 若两处皆写，不仅造成信息冗余，还会引发静态语法检查器抛出“重定义默认参数”的编译阻断。 |

> 在计算机编译原理中，函数调用时的参数匹配是基于位置（Positional Parameters）的，而不是基于名称（Named Parameters）。当编译器扫描到一行函数调用代码时，它的解析器（Parser）是从左到右处理括号内的表达式的。\
> C++ 的编译是以编译单元（通常是单个 .cpp 文件）为独立单位进行的。编译器在编译当前的 .cpp 文件时，根本不知道其他 .cpp 文件里写了什么。

示例：虚函数 + 默认参数（演示静态绑定）

```cpp
struct Base {
    virtual void log(int level = 1) { /* ... */ }
};
struct Derived : Base {
    void log(int level = 2) override { /* ... */ }
};

Base* p = new Derived;
p->log(); // 调用 Derived::log，但使用的是 Base 的默认参数 level = 1
```
上例说明：默认参数的值在编译期由指向对象的静态类型决定。


# 4.占位参数
占位参数（Placeholder Arguments / Unnamed Parameters）是指在 C++ 函数声明或定义中，只指定了数据类型、但没有给定变量名称的形参。

```cpp
void func(int a, int); // 第二个参数就是“占位参数”
```

在底层的机器码和汇编层面，占位参数和常规参数没有任何区别。
- 栈空间照常占用：在函数调用时，编译器依然会严格按照类型大小，将实参压入栈中（或放入寄存器）。
- 无法在函数体内访问：因为没有名字，函数体内部的代码无法通过任何手段获取或修改这个参数的值。它就像一个纯粹的“代码税”，调用时必须交，但函数内部不领情。

注意：仅在函数声明中可以省略参数名；在函数定义中若省略名称，则函数体内无法访问该参数。常见做法是在声明中使用占位参数以隐藏实现细节，而在定义中为需要使用的参数命名：

```cpp
// 头文件中的声明（占位参数）
void process(int, double);

// 源文件中的定义（需要使用时命名）
void process(int count, double value) {
    // 使用 count, value
}

// 如果在定义中也省略名称，则无法访问该参数：
void foo(int) { /* 不能使用该参数 */ }
```

| 维度 | 特性说明 | 典型代码示范 |
| :--- | :--- | :--- |
| **语法结构** | 仅保留形参的 `DataType`，直接省略 `VariableName`。 | `void alert(int, double);` |
| **物理本质** | 物理压栈开销完全保留，但函数体内绝对无法访问。 | 汇编级别与普通参数对齐。 |
| **首要功用** | 用于区分后置 `++` / `--` 运算符重载。 | `Type operator++(int);` |
| **工程价值** | 解决接口平滑重构、消除“未使用参数”编译警告、强制匹配第三方函数指针。 | `void onEvent(int id, string /*unused*/) {}` |
| **默认值联动**| 允许为占位参数赋予缺省值，实现编译期选择性忽略。 | `void log(string text, int = 1);` |


> 在 C++ 中重载 ++ 或 -- 运算符时，前置和后置的函数名都是 operator++。为了让编译器区分它们，C++ 规定用一个匿名的 int 占位参数来特指后置运算 【原理：编译器在底层会秘密翻译为 c.operator++(0)。这个传入的 0 就是为了填补 int 占位参数的坑】
