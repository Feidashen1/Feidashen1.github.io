# 类和对象

类（class）是对一类实体的抽象：它描述了该类型的状态（成员变量）和能转换这些状态的行为（成员函数）。可以把类看作对象的蓝图——定义了属性与操作的集合。

对象（object）是类的具体实例，拥有实际的存储空间和具体的状态值。对象通过类定义的接口对外提供行为：读取或修改状态、执行方法完成任务。

在 C++ 中，类既承担语义抽象，也与对象模型（内存布局、构造/析构、拷贝/移动语义）紧密相关。因此在设计类时，既要关注对外接口的清晰表达，也要注意资源管理与性能约束。

# 面向对象的三大特性
**封装（Encapsulation）**：将数据和操作数据的函数捆绑在一起，隐藏对象的具体实现细节。通过 public、private 等访问修饰符控制数据的权限。

**继承（Inheritance）**：允许在现有类的基础上创建新类，从而实现代码复用和层次结构。

**多态（Polymorphism）**：允许不同类的对象对同一消息做出不同的响应（通过虚函数和动态绑定实现）。

# 什么样的接口是“良好”的？
良好的接口应当为使用者提供一个清晰、稳定且易于理解的视图——屏蔽实现细节，只暴露完成任务所需的最小行为集合。这里的“块”通常是一个类或一组紧密相关的类；“用户”指其他开发者。

要点：
- 简洁的表面（small surface area）：公有 API 保持最小，减少学习成本与误用风险。
- 以“用户的词汇”表述：接口使用调用者熟悉的概念和类型，避免引入不必要的新抽象或术语。
- 隐藏可变实现：实现细节对外不可见，使得内部变更不会破坏依赖者。
- 明确语义与契约：文档化前置条件/后置条件、复杂度、异常与线程语义，令调用者易于推理与维护。
- 可组合与一致：遵循既有惯例（如 STL 风格、接受视图而非具体容器），便于与其它库组合使用。

 **总之，良好的接口让使用者用最少的知识、安全且高效地完成工作，同时为实现者保留演进空间。**

## 示例
下面给出 2–3 个简短的 C++ 示例，展示良好接口设计的一些常见做法。

- 显式构造，避免意外隐式转换：

```cpp
struct Foo {
    explicit Foo(int v) : value(v) {}
    int value;
};
// Foo f = 42; // 错误：需要显式构造 Foo(42)
```

- RAII 与所有权清晰（禁止拷贝、允许移动）：

```cpp
class Resource { /* ... */ };

class Guard {
    std::unique_ptr<Resource> res_;
public:
    explicit Guard(Resource* r) : res_(r) {}
    ~Guard() = default;
    Guard(const Guard&) = delete;
    Guard& operator=(const Guard&) = delete;
    Guard(Guard&&) noexcept = default;
    Guard& operator=(Guard&&) noexcept = default;
};
```

- 接受视图而非具体容器（可组合）：

```cpp
#include <span>
#include <vector>

void process(std::span<const int> data) {
    for (int x : data) {
        // ...
    }
}

int main() {
    std::vector<int> v = {1,2,3};
    process(v); // 支持 vector、数组等多种来源
}
```

# 封装是什么？
封装是把“可变的实现细节”和“稳定的接口”分离开来的设计策略。它像一道防火墙，阻止外部直接依赖内部实现，从而在内部改变时不会破坏外部代码。

关键思想：
- 可变的部分：实现细节（数据布局、辅助算法、私有方法等），这些可能随着需求或优化而改变。通常通过 `private`/`protected`、PIMPL、命名空间或模块边界来隐藏。
- 稳定的部分：接口——对外可见的 API（类的 `public` 成员、函数签名、模块暴露的类型）。这是其他代码与之交互的合同。

封装的好处：
- 降低耦合：外部只依赖接口，内部实现可自由演进。
- 降低维护成本：修复或优化实现不会强制改动依赖者。
- 限制误用：通过最小且明确的公有接口，减少错误使用的可能性。

实现手段（C++）：
- 访问控制：`private` / `protected` / `public`。
- PIMPL（Pointer to IMPL）隐藏头文件中的实现细节，减小编译依赖与二进制兼容问题。
- 以接口类或抽象基类定义稳定契约，具体实现放在实现文件或内部命名空间。

注意：封装是设计良好的手段，但它并不等同于安全（`封装 != 安全`）。封装主要用于降低耦合与提高可维护性，而不是防止恶意攻击或保证运行时安全。

# 良好的 C++ 接口
- 封装: 通过访问控制、private/protected 与抽象（纯虚函数或接口类）隐藏实现细节，暴露最少必需的行为。
- 最小表面: 公有 API 越小越好——减少组合复杂度与学习成本，避免不必要的成员函数和公有数据。
- 明确所有权语义: 使用值语义、std::unique_ptr/std::shared_ptr 或引用来清晰表达资源所有权；文档化谁负责释放资源。
- RAII 与资源管理: 资源由对象生命周期管理，构造成功即拥有资源，析构释放资源，保证异常安全。
- 异常安全与错误处理: 明确保证级别（无抛出 / 基本 / 强），对失败使用异常或 std::expected/std::optional 统一策略，避免未定义行为。
- 常量正确性: 对不修改状态的成员使用 const，对不会抛出的函数声明 noexcept（尤其是移动操作）。
- 显式转换与防止歧义: 使用 explicit 防止意外隐式转换，避免重载造成模糊的调用语义。
- 拷贝/移动语义清晰: 明确定义或禁用拷贝/移动构造与赋值（= default / = delete），保证效率与语义一致。
- 接口一致性与可组合性: 遵循 STL 风格（迭代器/范围、begin/end、free functions for symmetry），使用 std::span/string_view 接受范围/视图而非容器类型。
- 约束与表达能力: 在模板接口使用 concepts 或 requires 限定，提供有意义的编译期错误信息。
- 命名与文档: 使用描述性命名、简短注释与示例说明前置条件/后置条件/复杂度。
- 稳定性与封装边界: 若需要二进制兼容或隐藏实现细节，采用 PIMPL 或版本化策略。
线程语义明确: 明确说明对象是否线程安全和在并发场景下的使用限制。


# 关键字 struct 和 class 有什么区别？
struct 的成员默认是公有的，而类的成员默认是私有的。

    注意：应该明白地声明你的类成员为公有的、私有的、或者是保护的，而不是依赖于默认属性

struct 和 class 在其他方面是功能相当的。


# 什么是对象的句柄？它是指针吗？它是引用吗？它是指向指针的指针？它是什么？
“句柄”（handle）是对某个对象或资源的间接引用：它并不限定为某一种具体类型，而是一种抽象的标识或访问方式。

常见形式：
- 原始指针（如 `T*`）：直接指向对象所在位置。
- 整数或字符串 ID：作为键在表/数组/数据库中查找对象。
- 指向指针（如 `T**`）：增加一层间接以便对象可移动时仍可更新指向。
- 智能指针或封装类：将查找、生命周期和访问控制封装在类型内（常重载 `operator->`/`operator*`）。

为何使用句柄？
- 抽象位置和表示：调用者无需关心对象如何存储或定位，便于实现演进（例如从数组索引切换到哈希表）。
- 支持移动与重定位：通过间接层，可以在不通知所有使用方的情况下移动对象或改变存储策略。
- 集中管理生命周期：句柄的实现可以跟踪引用计数、延迟加载、缓存或回收策略，从而降低误用风险。

注意事项：
- 句柄是抽象，不同实现有不同性能与语义（例如是否可能失效、是否线程安全）。
- 要明确句柄的语义：谁拥有对象、何时可能失效、是否可序列化等。

小结：句柄不是特定的语言实体，而是设计上的工具——可以是指针、索引、ID 或封装类型，关键是它为对象访问提供了间接层和稳定的契约。
