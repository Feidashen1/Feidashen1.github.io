## 1.构造与析构概述

在 C++ 中，构造函数（Constructor）与析构函数（Destructor）是类的核心成员，负责对象的初始化和资源清理，共同保证对象生命周期的安全性。

从底层视角（汇编、编译原理与 C++ ABI 规范）来看，构造函数与析构函数本质上是特殊的类成员函数，由编译器在幕后进行代码插桩（Code Injection）与内存绑定。它们的底层实现机制可以从以下四个维度进行拆解：

### 1.1 this 指针与内存布局的绑定

在底层，构造函数和析构函数并不负责“分配”或“释放”对象本身的内存，它们只负责操作该内存块的内容 \
**隐式传递 this 指针**：无论是通过栈分配还是 new 堆分配，编译器在调用构造/析构函数时，都会将该对象的首地址作为第一个隐式参数（通常通过 ECX/RCX 寄存器或直接压栈）传递给函数，这就是底层 this 指针的来源。

### 1.2 编译器的“黑箱插桩”

构造与析构函数体，在编译后会被编译器强制“强插”许多前置和后置逻辑

> **构造顺序：**
> 1. 调用基类构造函数：如果有父类，首先调用父类构造。
> 2. 初始化虚函数表指针（vptr）：如果类含有虚函数，编译器在此处将当前类的虚函数表（vtable）地址写入到对象内存的前 4/8 个字节中。\
> 3. 调用成员变量的构造函数：按照成员变量在类中的声明顺序（而非初始化列表的顺序）依次调用它们的构造函数。\
> 4. 执行用户自己的代码：最后才执行在花括号 {} 里面写的代码；



> **析构顺序：**
> 1. 执行用户自己的代码：先执行你写在 {} 里的资源释放代码（如 delete ptr;）。
> 2. 调用成员变量的析构函数：按声明的逆序销毁各个成员变量。
> 3. 重置虚函数表指针（vptr）：将其指向当前类的虚表（防止在析构中调用子类的虚函数导致纯虚函数调用错误）。
> 4. 调用基类的析构函数：最后调用父类的析构函数

### 1.3.符号生成

根据标准的 Itanium C++ ABI 规范，编译器在编译一个构造或析构函数时，为了应对不同的调用场景，实际上会在二进制文件中生成多个不同名字的汇编函数符号

- C1 (Complete object constructor)：完整对象构造函数。负责初始化包含虚拟基类在内的整个对象。
- C2 (Base object constructor)：基类对象构造函数。当该类作为其他类的子类被构造时调用，它会跳过虚拟基类的初始化（因为虚拟基类由最底层的子类统一初始化）。
- D0 (Deleting destructor)：删除析构函数。配合 delete 关键字使用，它会先调用 D1 析构对象，然后调用 operator delete() 释放内存块。
- D1 (Complete object destructor)：完整对象析构函数。用于栈对象或通过类名直接销毁，只做清理不做内存释放。
- D2 (Base object destructor)：基类对象析构函数。子类析构时调用它来清理父类部分，同样不释放总内存。

### 1.4.虚析构函数与动态绑定
> 为什么基类的析构函数必须是虚函数？这也是高频的底层面试题

```cpp
Base* ptr = new Derived();
delete ptr;
```

**如果 ~Base() 不是虚函数**：汇编层面的 delete ptr 会直接静态绑定（Static Binding）到 Base::D0 符号上。编译器只知道它是个 Base，所以只会执行 Base 的清理，导致 Derived 子类中特有的成员变量（如动态申请的数组）发生内存泄漏。

**如果 ~Base() 是虚函数**：对象的首地址处会存在一个 vptr（虚表指针）。当执行 delete ptr 时，底层汇编会通过类似 mov rax, [rcx]; call [rax + offset] 的指令，去虚函数表中查找真实的析构函数入口。此时它找到的是 Derived 的析构函数（D0），从而能够从子类到父类完整、正确地释放所有资源


## 2. 虚函数表（vtable）在构造和析构执行过程中的动态切换

**虚函数表指针（vptr）在构造和析构过程中的动态切换，是编译器为了保证程序安全而设计的核心机制**

在构造和析构执行期间，对象的“身份”一直在发生动态的蜕变，vptr 也在高频地切换指向

> 为什么必须动态切换？\
> 在标准 C++ 中有一条核心安全性规则：在执行基类的构造或析构函数时，对象中属于派生类的成员变量要么尚未初始化（构造时），要么已经被销毁（析构时）。因此，在基类构造/析构中调用虚函数，绝对不能路由到子类的实现，只能调用当前类或其更上层基类的虚函数。\
> 为了严格实现这一规则，编译器必须在进入每个类的构造/析构函数时，强行改写对象的 vptr。

### 2.1 vptr 切换细节
假设有三个类，存在继承关系：`Base -> Middle -> Derived`。每个类都重写了虚函数 `foo()`。

#### 2.1.1 构造
当执行 Derived* d = new Derived(); 时 底层的内存与 vptr 切换时序如下：

阶段一：原始内存（分配阶段）\
operator new 分配了一块干净或充满垃圾数据的内存。此时，对象内存的前 4/8 字节（vptr 所在位置）为 随机的垃圾位

阶段二：进入 Base 构造 \
编译器首先调用 Base 的构造函数。\
底层动作：在执行 Base 构造函数体（用户代码）之前，编译器的前置插桩代码会执行一条汇编指令：`assemblymov [rcx], offset vtable for Base`  将 vptr 指向 Base 的虚表\
此时状态：如果此时在 Base 的构造函数里调用 foo()，通过 vptr 查找，执行的是 Base::foo()

阶段三：进入 Middle 构造 \
Base 构造执行完毕，控制权回到 Middle 构造函数\
底层动作：在进入 Middle 用户代码前，编译器再次强行修改 vptr：`mov [rcx], offset vtable for Middle`  改写 vptr 指向 Middle 的虚表\
此时状态：此时对象身份变成 Middle。如果调用虚函数，执行的是 Middle::foo()。

阶段四：进入 Derived 构造\
最后进入最终派生类 Derived 的构造函数。\
底层动作：编译器最后一次改写 vptr：`assemblymov [rcx], offset vtable for Derived` 改写 vptr 指向最终的 Derived 虚表\
此时状态：构造彻底完成，对象正式完全“活了过来”，身份定格为 Derived



#### 2.1.2 析构

析构过程是构造的完全逆演。当执行 delete d; 时，vptr 经历的是一个剥离身份的过程：

1. 执行 ~Derived()：此时 vptr 依然指向 Derived 虚表。执行完用户自定义的析构代码后，在销毁成员变量之前，编译器通常不需要立即改写 vptr，因为此时依然是 Derived 的生命周期末尾。

2. 进入 ~Middle()：在控制权交给 Middle 的析构函数体之前，编译器插桩代码会立刻重写 vptr：`assemblymov [rcx], offset vtable for Middle` 强行将身份降级为 Middle \
为什么？ 因为此时 Derived 特有的成员变量已经被销毁了！如果不降级，万一在 ~Middle() 中间接调用了虚函数，就会访问到已经被销毁的 Derived 成员，导致崩溃。

3. 进入 ~Base()：同样，在进入 Base 析构函数体前，vptr 被最后一次改写：`assemblymov [rcx], offset vtable for Base`   ; 身份降级为 Base

4. 内存释放：所有析构函数执行完毕，内存重新变回无意义的随意位，最后被 operator delete 彻底回收。


#### 2.1.3 纯虚函数调用（Pure Virtual Call）问题

程序在运行期间，通过一个虚函数表指针（vptr）访问了虚函数表（vtable），而该表项中的函数指针正处于“未解决的空槽（Null/Error Slot）”状态，导致了奔溃

```cpp
class AbstractBase {
public:
    AbstractBase() { 
        // 危险：在构造函数中调用了虚函数
        log(); 
    }
    virtual void log() = 0; // 纯虚函数
};

class Concrete : public AbstractBase {
public:
    virtual void log() override { std::cout << "Concrete log"; }
};
```

底层发生了什么？
1. 当创建 Concrete 对象时，首先会进入 AbstractBase 的构造函数。根据前文原理，编译器在此时将对象的 vptr 指向了 AbstractBase 的虚表。
2. 关键在于：AbstractBase 的虚表中，log 这一项对应的函数指针是 0（或指向一个特殊的报错函数 __cxa_pure_virtual），因为它是纯虚函数，没有实现。
3. 构造函数内的 log() 被触发，汇编指令去读取 vptr 并跳转到这个报错入口。
4. 结果：程序直接崩溃，操作系统抛出 pure virtual method called 异常。


> 在底层的视角里，vptr 不是一个一成不变的指针，而是一个动态的身份同步器。编译器通过在构造/析构的每一个入口处高频插入 `mov [this], vtable_address` 指令，确保了对象在任何一个生命周期切面上，其虚函数行为都与其当前已经被初始化的安全边界严格契合。

#### 2.1.4 纯虚函数
纯虚函数（Pure Virtual Function） 是一种特殊的虚函数，它的核心特征是在声明时以 = 0 结尾，且通常在基类中不提供具体实现

从设计模式和底层机制来看，纯虚函数的核心作用可以概括为两点：定义强约束的“标准接口” 和 强制将类变为“抽象基类”

**1. 强制制定接口规范（规范派生类行为）**
如果一个基类声明了纯虚函数，那么任何继承自该类的派生类必须实现（重写）这个函数。如果子类敢不实现，编译器就会无情报错。

    目的：确保所有子类都具备高度一致的对外接口，实现高层次的多态性。

**2.阻止基类被实例化（创建抽象基类）**
一旦一个类中包含至少一个纯虚函数，这个类就自动变成了抽象类（Abstract Class）。

    底层的强制限制：绝对无法直接创建这个类的对象


> **虚函数表（vtable）中的“空指针占位”**：对于普通虚函数，类的虚表里存放的是该函数的实际汇编地址。而对于纯虚函数，编译器在构建基类的虚表时，该函数对应的槽位（Slot）里存放的是 0（NULL指针），或者指向一个操作系统底层的报错桩函数（如 __cxa_pure_virtual）。

> **不允许实例化的底层原因**：正是因为基类虚表中存在这种“未解决的空指针槽位”，如果允许实例化基类对象，一旦调用该函数就会引发可怕的段错误（Segment Fault）。因此编译器在前端语义检查阶段就直接封死了创建基类对象的可能性。

**常见问题：纯虚函数绝对不能有实现吗？**

答案是：可以有。在 C++ 中，你可以为纯虚函数编写函数体：
```cpp
// 抽象基类：形状
class Shape {
public:
    // 1. 声明纯虚函数，尾部加 "= 0"
    virtual void draw() = 0; 
    virtual double getArea() = 0;

    // 虚析构函数（底层必备，防止内存泄漏）
    virtual ~Shape() {} 
};

// 派生类：圆形
class Circle : public Shape {
private:
    double radius;
public:
    Circle(double r) : radius(r) {}

    // 2. 子类必须实现基类的所有纯虚函数
    void draw() override { std::cout << "画一个圆形" << std::endl; }
    double getArea() override { return 3.14 * radius * radius; }
};


// 实现纯虚函数
void Shape::draw() {
    std::cout << "基类的默认绘制逻辑";
}
```
请谨慎使用此类代码。为什么还要写 = 0？：写了 = 0 就意味着子类依然必须显式重写它，不重写就报错。这有什么用？：子类重写时，可以通过 Shape::draw(); 来调用基类的这份“默认实现”，从而实现代码复用，同时又保留了接口的强制约束力。


总结不同：

1. 普通虚函数：提供默认实现，子类“可以”重写（侧重于代码复用）。
2. 纯虚函数：不提供（或仅提供可选的）默认实现，子类“必须”重写（侧重于接口规范）。
