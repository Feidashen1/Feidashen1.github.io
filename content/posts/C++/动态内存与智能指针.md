## 1.动态内存

C++ 程序中的内存分为两个部分：

栈：在函数内部声明的所有变量都将占用栈内存。

堆：这是程序中未使用的内存，在程序运行时可用于动态分配内存。

### 1.1 程序对象的生存期

- 全局对象在程序启动时分配，在程序结束时销毁。
- 对于局部自动对象，进入其定义所在的程序块时被创建，在离开块时销毁。
- 局部 static 对象在第一次使用前分配，在程序结束时销毁。
- **C++还支持动态分配对象。动态分配的对象的生存期与它们在哪里创建是无关的，只有当显式地被释放时，这些对象才会销毁。**

在C/C++程序中（线程）栈空间是有限的，大部分变量使用的都是动态分配来的堆内存，这些动态申请来的堆内存是需要开发者通过代码去自行管理的。

### 1.2 内存模型

内存模型：

![内存模型](../images/cpp_images/memory.png)

- 栈区（Stack）：函数局部变量、形参。编译器自动管理。速度极快。【栈区地址从高到低分配】
- 堆区（Heap）：由 new / malloc 动态分配的内存。开发者（或智能指针）手动管理。**用于存放进程运行中被动态分配的内存段，它的大小并不固定，可动态扩张或缩减。** 【堆区的地址是从低到高分配】
- 全局/静态存储区（Global / Static Space）：存储全局变量和 static 变量。程序启动时分配，结束时释放。初始化的全局变量和静态变量存放在一块区域(BSS区)，未初始化的全局变量和静态变量在相邻的另一块区域（data数据区）。
- 常量存储区（Constant Storage）：存储只读常量（如字符串字面量 "hello"）。不可修改，程序结束后由系统释放。
- 代码区（Code Segment / Text）：存储二进制机器指令。这部分区域的大小在程序运行前就已经确定，并且内存区域通常属于只读，某些架构也允许代码段为可写，即允许修改程序。在代码段中，也有可能包含一些只读的常数变量 ，例如字符串常量等。程序段为程序代码在内存中的映射。一个程序可以在内存中有多个副本。

### 1.3 new 和 delete

c++ 使用 new 和 delete 管理动态内存

- new：在堆中为对象分配空间并返回指向该对象的指针
- delete：接受一个动态对象的指针，销毁该对象并释放内存。


| 特性维度         | `new` / `delete`                         | `malloc` / `free`                     |
| :--------------- | :--------------------------------------- | :------------------------------------ |
| **本质属性**     | C++**运算符**（关键字，支持重载）        | C 标准库**函数**（`stdlib.h`）        |
| **生命周期管理** | 自动调用**构造函数 / 析构函数**          | 仅分配/释放内存，**不触发**构造/析构  |
| **类型安全**     | 返回具体类型指针，**不需要**强制类型转换 | 返回`void*`，**必须**进行强制类型转换 |
| **计算内存大小** | 编译器根据类型**自动计算**所需空间大小   | 必须手动通过`sizeof()` 传入字节数     |
| **分配失败行为** | 默认**抛出异常**（`std::bad_alloc`）     | 返回**`NULL`** 指针                   |
| **数组处理**     | 有专门的`new[]` 和 `delete[]` 配对       | 需手动计算总大小（或使用`calloc`）    |
| **内存扩容**     | 不支持直接扩容（需重新分配并拷贝）       | 可以使用`realloc` 在原有基础上扩容    |
| **相互重构性**   | 可以通过`set_new_handler` 处理内存不足   | 无法干预其内部的内存分配行为          |

可能会遇到的问题点：
1. 使用了 new[] 却用 delete 释放。这会导致只有数组第一个元素的析构函数被调用，且堆内存管理器的计数器错乱，引发程序直接崩溃或内存泄漏；
2. 执行 delete ptr; 后，ptr 变量本身还指向那块已经被释放的地址。后续如果误用 ptr 会导致未定义行为；
3. 在 new 之后、delete 之前，如果程序因为 if 分支提前 return，或者抛出了异常，导致 delete 没被执行，该块堆内存将永久泄漏；

忘记释放内存会引起内存泄漏，释放了后继续引用指针会引用非法内存。如果忘记释放内存，在程序结束时会由操作系统自动回收。

> 静态内存是由系统自动分配内存，由系统自动释放。静态内存是在栈中分配的，假如main函数调用另一个函数，那么就把被调用函数压到一个栈里面。执行这个被调函数就是使系统为这个函数分配的所有内存空间逐个出栈。出栈全部结束就是被调用函数执行完毕。
> 动态内存是由程序员手动释放，函数终止不会被系统自动释放。这说明他肯定不是在栈里面分配的。那他是在什么地方分配的呢？是在“堆中”分配的，栈是一种存储结构，堆不是一种存储结构，堆是分配内存的一种排序方式。也就是说，动态内存是以堆排序的方式分配的。



## 2.智能指针
在 C++11 及以后的现代标准中，不鼓励直接编写 new 和 delete。强烈建议使用标准库的智能指针来托管内存，它们利用 RAII 机制，在超出作用域时自动调用 delete。

新标准库提供两种智能指针和一个伴随类管理动态内存，都定义在头文件 memory 中：
- shared_ptr：允许多个指针指向一个对象
- unique_tpr：独占所指的对象
- weak_ptr：一种弱引用，指向 shared_ptr 所管理的对象

要注意到：**智能指针实际上是一个类模板**。


### 2.1 shared_ptr

`std::shared_ptr<T>` 是一个类模板，它的对象行为像指针，但是它还能记录有多少个对象共享它管理的内存对象。

```cpp
element_type*    _M_ptr;         // Contained pointer.
__shared_count<_Lp>  _M_refcount;    // Reference counter.
```
std::shared_ptr 在内部维护一个引用计数，其只有两个指针成员，**一个指针是所管理的数据的地址**；还有**一个指针是控制块的地址**，包括引用计数、weak_ptr计数、删除器(Deleter)、分配器(Allocator)。

```
+------------------------------------------+
|               控制块 (Control Block)      |
+------------------------------------------+

| 1. Strong Ref Count (强引用计数)          | -> 决定对象何时被析构 (~T)
| 2. Weak Ref Count   (弱引用计数)          | -> 决定控制块何时被销毁 (free)
| 3. Custom Deleter   (自定义删除器, 可选)  | -> 用于非 delete 释放(如 fclose)
| 4. Custom Allocator (自定义分配器, 可选)  | -> 用于控制块自身的内存分配
+------------------------------------------+
```
- **Strong Ref Count (强引用计数)**：记录当前有多少个 std::shared_ptr 指向该对象。
- **Weak Ref Count (弱引用计数)**：记录当前有多少个 std::weak_ptr 指向该对象。注意：在标准库实现中，如果强引用计数大于 0，弱引用计数通常会隐式 +1（作为强引用的占位符号）。



因为不同shared_ptr指针需要共享相同的内存对象，因此引用计数的存储是在堆上的。而unique_ptr只有一个指针成员，指向所管理的数据的地址。**因此一个shared_ptr对象的大小是raw_pointer大小的两倍。**

多个 `std::shared_ptr<T>` 可以共享同一个对象。当最后一个 `std::shared_ptr<T>` 被销毁时，它会自动释放它所指向的对象。

一个`shared_ptr<T>`指针可以通过`make_shared<T>`函数来创建，也可以通过拷贝或赋值另一个`shared_ptr`来创建：

```cpp
shared_ptr<int> p;              // 默认初始化为空指针
shared_ptr<int> p(q);           // q 也是一个 shared_ptr,p 是 q 的拷贝，此操作会递增 q 中的计数器。
shared_ptr<int> p(qnew);        // qnew 是一个指向动态内存的内置指针（qnew = new int;)）
shared_ptr<int> p(u);           // u 是一个 unique_ptr。p 从 u 接管了对象的所有权，u 被置为空
shared_ptr<int> p(q, deleter);  // q 是一个内置指针。p 将使用可调用对象 deleter 来代替 delete
shared_ptr<int> p(p2, deleter); // p2 是一个 shared_ptr，p 是 p2 的拷贝，唯一的区别是 p 将可调用对象 d 来代替 delete。
 
auto p = make_shared<int>(10);  //返回一个 shared_ptr，指向一个初始化为 10 的动态分配的 int 对象。注意不同于 make_pair
```

> **make_shared 与 new 的内存布局差异** \
> \
> 创建 shared_ptr 有两种主流方式，它们的控制块内存布局完全不同：
> 1. **方式一：std::shared_ptr<T>(new T())（分立内存）**：auto p = std::shared_ptr<Widget>(new Widget()); \
> 内存布局：在堆上分配两次内存。第一次：new Widget() 分配一块内存。第二次：智能指针内部为控制块单独分配一块内存。\
> 缺点：两次内存分配效率较低；**两个内存块不连续**，对 CPU 缓存（Cache）不友好；
> 2. **方式二：std::make_shared<T>()（合并内存）**：auto p = std::make_shared<Widget>()：auto p = std::make_shared<Widget>(); \
> 内存布局：在堆上仅分配一次内存。编译器会计算出 Widget 大小与控制块大小的总和，打包申请一块连续的内存。\
> 优点：效率高，节省一次堆分配开销；内存连续，对 CPU 缓存极度友好。\
> 缺点（大坑）：由于内存是一体的，必须等到“阶段 2”（弱引用计数也清零）时，整块合并的内存才会被统一释放。如果对象很大，且 weak_ptr 一直存在，那么即使对象早就析构了，它占用的那部分大内存也无法归还给系统。 


#### 2.1.1 shared_ptr 操作

```cpp
sp              // 智能指针作为 if 的判断条件时检查其是否为空，若 sp 指向一个对象，则为 true
sp->mem;        // 等价于 (*p).mem。用于当 sp 指向一个类时
sp.get();       // 返回 sp 中保存的指针。要小心使用！
swap(p, q);     // 交换 p 和 q 中的指针
p.swap(q);      // 同上
 
p = q;          // 此操作会递增 q 中的计数器，递减 p 原来的计数器，若其变为 0，则释放。
p.unique();     // 若 p.use_count() 为 1，返回 true，否则返回 false
p.use_count();  // 返回与 p 共享对象的智能指针数量。可能运行很慢，主要用于调试
 
p.reset();     // 将 p 置为空，如果 p 计数值为 1，释放对象。
p.reset(q);    // q 是一个内置指针，令 p 指向 q。
p.reset(q, d); // 调用可调用对象 d 而不是 delete 来释放 q
```

**不要使用 get 初始化另一个智能指针或为智能指针赋值**：shared_ptr 的关联计数只应用于自己的拷贝，**如果使用某智能指针的 get 函数初始化另一个智能指针，两个指针的计数是不关联的**，销毁一个就会直接释放内存使另一个成为空悬指针。

> 调用 .get() 时，它仅仅返回堆内存的原始地址，而不会转移所有权，也不会共享引用计数
```cpp
#include <memory>

void danger_zone() {
    // 1. 创建第一个智能指针，它独自拥有该堆内存
    std::unique_ptr<int> ptr1 = std::make_unique<int>(42);

    // 2. 错误操作：通过 get() 获取原始指针，并用它初始化第二个智能指针
    int* raw_ptr = ptr1.get();
    std::unique_ptr<int> ptr2(raw_ptr); // 致命错误！
    
    // 或者 shared_ptr 的错误混用：
    // std::shared_ptr<int> sptr1 = std::make_shared<int>(100);
    // std::shared_ptr<int> sptr2(sptr1.get()); // 致命错误！

} // 函数结束，ptr2 先析构，释放堆内存。接着 ptr1 析构，再次释放同一块内存 —— Double Free 崩溃！
```

ptr1 认为自己是这块内存的唯一主人。ptr2 也是独立创建的，它也认为自己是这块内存的唯一主人。两个指针互不相知，在作用域结束时，它们会对同一个内存地址分别调用一次 delete。


**既然 .get() 这么危险，为什么标准库还要提供它** 它的唯一正确用途是：向下兼容只需要原始指针的旧式 C/C++ 接口。

## 2.2 unique_ptr
同一时刻只能有一个 unique_ptr 指向一个给定对象。当 unique_ptr 被销毁时，指向对象也被销毁。定义 unique_ptr 时，需要绑定到一个 new 返回的指针上（不同于 shared_ptr）。

因为 unique_ptr 独有它指向的对象，所有它不支持拷贝和赋值操作。，对应拷贝构造函数和赋值运算符函数已定义删除。

```cpp
unique_ptr<int> u1;             // 定义一个空 unique_ptr
unique_ptr<int> u1(new int());  // 正确
unique_ptr<int,deleter> u;      // 定义一个空 unqiue，用可调用对象 deleter 代替 delete
unique_ptr<int,deleter> u(d);   // 空 unique，用类型为 deleter 的对象 d 代替delete
unique_ptr<int,decltype(d)*> u(new int(42),d);
 
unique_ptr<int> u2(u1);         // 错误：不支持拷贝
```


### 2.2.1 uniqe_ptr 操作
```cpp
u.get();
u1.swap(u2);swap(u1,u2);
 
u = nullptr;             // 释放 u 指向的对象并将 u 置为空
auto u2 = u.release();   // u 放弃对指针的控制权，返回 u 中保存的内置指针，并将 u 置为空，注意 u2 的类型是内置指针，而不是 unique_ptr
u.reset();               // 释放 u 指向的对象
u.reset(nullptr);        // 释放 u 指向的对象，并将 u 置为空，等价于 u = nullptr;
u.reset(q);              // 令 u 指向内置指针 q 指向的对象
```

可以通过 release 或 reset 将指针的所有权从一个 unique_ptr 转移给另一个 unique。

```cpp
unique_ptr<int> u2(u1.release());  // 控制器转移给 u2，u1 置为空
u3.reset(u1.release());            // 释放 u3 原来指向的内存，u3 接管 u1 指向的对象。
```


> release 返回的指针不能空闲，必须有其他指针接管对象。如果是一个内置指针接管了 release 返回的指针，那么程序就要负责资源的释放。

```cpp
u.release();             // 错误：release 不会释放内存，没有其他指针接管内存。
auto u2 = u1.release();  // 正确，但必须记得 delete p
```

std::move() 可以将一个unique_ptr转移给另一个unique_ptr或者shared_ptr。转移后，原来的unique_ptr将不再拥有对内存的控制权，将变为空指针。

```cpp
std::unique_ptr<int> p1 = std::make_unique<int>(0);
std::unique_ptr<int> p2 = std::move(p1); 
// now, p1 is nullpt
```

> `auto_ptr` 是标准库的较早版本包含的一个类，它具有 unique_ptr 的部分特性。相比于 unique_ptr，不能在容器中保存 auto_ptr，也不能从函数返回 auto_ptr。

## 2.3 weak_ptr
标准库还定义了一个名为weak_ptr的伴随类，它是一种**弱引用**，指向shared_ptr所管理的对象，而不影响所指对象的生命周期，也就是将一个weak_ptr绑定到一个shared_ptr不会改变shared_ptr的引用计数。不论是否有weak_ptr指向，一旦最后一个指向对象的shared_ptr被销毁，对象就会被释放。




### 2.3.1 weak_ptr 操作
weak_ptr对它所指向的shared_ptr所管理的对象没有所有权，不能对它解引用，因此若要读取引用对象，必须要转换成shared_ptr。 C++中提供了lock函数来实现该功能。如果对象存在，lock()函数返回一个指向共享对象的shared_ptr，否则返回一个空shared_ptr。

```cpp
if(shared_ptr<int> np = wp.lock())  // 如果 np 不为空则条件成立
```

因为 weak_ptr 的对象可能被释放的，因此不能直接访问对象，必须调用 lock()。lock() 检查 weak_ptr 所指的对象是否仍存在，如果存在，返回一个指向共享对象的 shared_ptr。

**返回的这个 shared_ptr 会使引用计数加 1。**

```cpp
w = p;            // p 可以是一个 shared_ptr 或 weak_ptr。赋值后 w 与 p 共享对象。
w.reset();        // 将 w 置为空
w.use_count();    // 返回与 w 共享对象的 shared_ptr 的数量
w.expired();      // 若 w.use_count() 为 0，返回 true，否则返回 false。expired 是 “过期的” 意思
w.lock();         // 如果 w.expired 为 true，返回一个空 shared_ptr；否则返回一个指向 w 的对象的 shared_ptr
```

weak_ptr提供了一个成员函数expired()来判断所指对象是否已经被释放。如果所指对象已经被释放，expired()返回true，否则返回false。
```cpp
if(!wp.expired())
```

### 2.3.2 引入 weak_ptr 的原因
循环引用（Circular Dependency）是指两个或多个对象之间通过shared_ptr相互引用，形成了一个环，导致它们的引用计数都不为0，从而导致内存泄漏。

如果只有 std::shared_ptr，在遇到循环引用时，会导致致命的内存泄漏。
```cpp
#include <iostream>
#include <memory>

class B; // 前置声明

class A {
public:
    std::shared_ptr<B> ptrB; // A 持有 B 的 shared_ptr
    ~A() { std::cout << "A 被销毁\n"; }
};

class B {
public:
    std::shared_ptr<A> ptrA; // B 持有 A 的 shared_ptr
    ~B() { std::cout << "B 被销毁\n"; }
};

void leaky_code() {
    auto a = std::make_shared<A>();
    auto b = std::make_shared<B>();
    
    a->ptrB = b; // b 的引用计数变为 2
    b->ptrA = a; // a 的引用计数变为 2
} // 函数结束，a 和 b 的局部变量析构，引用计数减 1，但各自仍为 1。
  // 导致 A 和 B 的析构函数永远不会被调用，堆内存永久泄漏！
```

将其中一个改为 std::weak_ptr只需要把 class B（或 A）内部的指针改为 std::weak_ptr，由于它不增加引用计数，当函数结束时，对象就能被正常销毁：

```cpp
class B {
public:
    std::weak_ptr<A> ptrA; // 破解核心：改为弱引用，不增加 A 的引用计数
    ~B() { std::cout << "B 被销毁\n"; }
};
```

### 2.3.3 weakptr 的应用

**1.破解循环引用**：在复杂的对象图（Object Graph）中，如果两个或多个对象相互使用 std::shared_ptr，会导致谁都无法释放。

场景 1：树/图（Tree/Graph）结构
- 设计原则：父节点拥有子节点，子节点观测父节点。
- 实现：父节点使用 std::vector<std::shared_ptr<Node>> 指向子节点；子节点使用 std::weak_ptr<Node> 反向指向父节点。如果子节点也用 shared_ptr 就会导致无法释放。

场景 2：主从模式（Master-Worker）
- Master 统一管理多个 Worker（强引用）。Worker 需要随时向 Master 汇报状态（弱引用）。

**2.线程安全的对象缓存区（Cache）**：实现一个单例或全局的缓存池时，std::weak_ptr 是最佳选择。
- 业务诉求：多个业务模块如果请求同一个资源（如 map_level1），希望能复用同一份内存。当所有业务模块都不再使用这个资源时，内存应该立刻释放，缓存区不需要强行把大资源留在内存中。
- 实现机制：缓存区的 std::map 中存储的是 std::weak_ptr<Resource>。当模块 A 请求资源时，缓存区通过 lock() 检查：如果提升成功（返回 shared_ptr），说明资源还在，直接复用。如果提升失败（返回 nullptr），说明之前的所有使用者都释放了它，缓存区此时重新加载该资源，并更新 weak_ptr。

**3.观察者模式（Observer Pattern）**：在发布-订阅（Publish-Subscribe）或观察者模式中，被观察者（Subject）需要维护一个观察者（Observer）列表。
- 坑：如果 Subject 强引用（shared_ptr）所有 Observer，那么只要 Subject 活着，Observer 就永远无法被销毁，即便 Observer 的宿主界面或业务已经关闭了。这会导致严重的内存泄漏与“僵尸对象”响应。
- Subject 内部使用 std::vector<std::weak_ptr<Observer>> 维护通知列表。当事件发生时，Subject 遍历列表并尝试 lock() 提升：提升成功：调用 observer->onEvent()。提升失败：说明该观察者已被外部销毁，Subject 顺便将这个死掉的 weak_ptr 从列表中彻底移除（清理门户）。

**异步回调（Asynchronous Callbacks）与多线程安全**：例如在现代 C++ 多线程编程中（例如使用 asio 或网络通信库），经常会发出一个异步请求，并绑定一个回调函数。
- 隐患：网络请求需要 5 秒才返回。在这 5 秒内，发起请求的对象（比如一个 UI 弹窗类 MyDialog）已经被用户手动点击关闭并销毁了。5 秒后网络数据传回，回调函数被触发，如果回调内部直接访问已经死亡的 MyDialog 指针，就会导致 悬空指针访问（Use-After-Free）崩溃。

## 3 allocator 类
std::allocator 是 C++ 标准库提供的一个内存分配器模板类。它是所有标准库容器（如 std::vector, std::list, std::map 等）底层的默认内存管理器。

在常规开发中，我们使用 new 动态创建对象。但正如前面讨论的，new 会同时做两件事：
1. 在堆上分配足够大小的原始内存。
2. 立即调用该类型的构造函数初始化对象。

new 的局限性是它将内存分配和对象构造结合在了一起，对应的 delete 将对象析构和内存释放结合在了一起。

这种紧密绑定在编写高性能容器（如 std::vector）时会带来巨大的性能浪费：\
**问题**：假创建了一个容量为 100 的 vector，但目前只插入了 3 个元素。如果用 new，编译器必须一口气调用 100 次默认构造函数。\

如果 `std::vector` 的扩容（`reserve(100)`）底层直接写成 `data = new Widget[100];`，编译器在执行这行代码时，在底层强制绑定了两个动作：
- 动作 A（内存分配）：在堆上开辟连续的、能容纳 100 个 Widget 对象的字节空间。
- 动作 B（强制构造）：立刻、自动在这些空间上调用 100 次 Widget 的默认构造函数。

**解法**：std::allocator 允许我们先申请 100 个对象的原始内存空間（不触发构造） ，等到真正插入数据时，再在指定的内存位置上按需调用构造函数。

标准库 allocator 类定义在头文件 memory 中，可以实现内存分配与对象构造的分离。

### 3.1 allocator 类 使用


现代 C++ 中，std::allocator 的直接使用需要配合 std::allocator_traits。它将动态对象的生命周期精细拆分为四个阶段：

    1. allocate()   --> 2. construct() --> 3. destroy() --> 4. deallocate()
    仅申请原始内存 --> 在内存上构造对象 --> 调用析构函数 --> 释放原始内存

- allocate(n)：分配一段能够保存 n 个特定类型对象的连续原始内存，返回首地址指针（不调用构造函数）。
- deallocate(p, n)：释放从指针 p 开始、容量为 n 个对象的原始内存（不调用析构函数）。
- std::allocator_traits<Alloc>::construct(alloc, p, args...)：在指针 p 指向的未初始化内存上，利用 args 参数调用构造函数（底层基于 placement new）。
- std::allocator_traits<Alloc>::destroy(alloc, p)：调用指针 p 指向对象的析构函数，将对象销毁，但不释放内存。

针对上面容器的问题解决示例：

```cpp
#include <iostream>
#include <memory>  // 包含 std::allocator 和 allocator_traits

class Widget {
public:
    Widget(int x) : data(x) { std::cout << "Widget 构造: " << data << "\n"; }
    ~Widget() { std::cout << "Widget 析构: " << data << "\n"; }
private:
    int data;
};

int main() {
    // 1. 定义分配器，显式指定管理的类型
    std::allocator<Widget> alloc;
    using Traits = std::allocator_traits<std::allocator<Widget>>;

    size_t capacity = 3;
    
    // 2. 阶段一：仅分配 3 个 Widget 大小的堆内存（此时完全没有调用构造函数！）
    Widget* buffer = alloc.allocate(capacity);
    std::cout << "--- 内存分配完毕，此时尚未构造对象 ---\n";

    // 3. 阶段二：按需构造对象（模拟向 vector 插入 2 个元素）
    Traits::construct(alloc, buffer, 100);     // 在第 0 个位置构造
    Traits::construct(alloc, buffer + 1, 200); // 在第 1 个位置构造
    std::cout << "--- 2 个对象构造完毕，还有 1 个格子保持原始内存状态 ---\n";

    // 4. 阶段三：手动销毁已构造的对象（必须严格按照构造的数量来销毁）
    Traits::destroy(alloc, buffer);
    Traits::destroy(alloc, buffer + 1);
    std::cout << "--- 对象析构完毕，但堆内存依然保留 ---\n";

    // 5. 阶段四：彻底释放这块堆内存，归还给系统
    alloc.deallocate(buffer, capacity);
    std::cout << "--- 堆内存彻底释放 ---\n";
}
```

construct 和 destory 一次都只能构造或销毁一个对象，要想完成对所有元素的操作，需要通过指针来遍历对每个元素进行操作。

传递给 deallocate 的 p 必须指向由 allocate 分配的内存，大小参数 n 必须与 allocate 分配内存时提供的大小参数一样。    

除了使用 construct 构造对象外，标准库还提供了两个伴随算法，定义在头文件 memory 中，他们在给定的位置创建元素。
```cpp
uninitialized_copy(b, e, b2);     // 从迭代器 b 和 e 指定的输入范围中拷贝元素到从迭代器 b2 开始的未构造的原始内存中。b2 指向的内存需要足够大。
uninitialized_copy_n(b, n, b2);   // 从迭代器 b 指向的元素开始，拷贝 n 个元素到 b2 开始的内存中。
uninitialized_fill(b, e, t);      // 在 b 和 e 指定的范围内创建对象，对象均为 t 的拷贝
uninitialized_fill_n(b, n, t);    // 在从 b 开始的 n 个位置创建对象，对象均为 t 的拷贝。
```

uninitialized_copy 函数返回指向构造的最后一个元素之后位置的指针。

比如希望将一个 vector 拷贝到动态内存中，并对后一半空间用给定值填充：
```cpp
allocator<int> alloc;
auto p = alloc.allocate( v.size() * 2 );
auto q = uninitialized_copy( v.begin(), v.end(), p );
uninitialized_fill_n( q,v.size(),42);
```
