C++的继承允许在保持原有基类（父类）特性的基础上进行功能扩展，实现代码重用；多态则通过虚函数实现动态绑定，允许使用基类指针或引用调用派生类（子类）的特定实现，从而达到“同一接口、多种形态”的效果。

继承是面向对象编程和抽象数据类型（ADT）编程的区分标志

## 1.类的类型转换
将派生类（子类）的指针转换成基类（父类）的指针，在 C++ 中是一种非常基础且常用的操作，通常被称为 **向上转型（Upcasting）**。

转换的两种形式：
1. 隐式转换：如果派生类是通过 public 继承基类的，编译器允许直接进行隐式转换；
2. 显式转换：为了使代码可读性更高，或者在处理特殊的非公有继承（如 protected 或 private 继承）时，可以使用强转。

- static_cast：用于 public 继承的向上转型，效果与隐式转换相同。
- dynamic_cast：通常用于向下转型，但在向上转型时也适用，效率略低，因为会有运行时类型检查。

当派生类指针转换成基类指针后，通过该基类指针访问成员时，**行为取决于函数是否为虚函数（virtual）**：
- 访问普通成员函数：调用的是基类的函数（静态绑定）。
- 访问虚函数：调用的是派生类重写后的函数（动态绑定，即多态）。
- 无法访问派生类特有成员：基类指针无法感知子类中新定义的属性和方法。

> 必须是公有（public）继承：如果使用 private 或 protected 继承，基类指针无法在类外直接指向派生类对象，隐式转换会报错

## 2.public, private 和 protected 与 友元
public：只要能拿到这个类的对象，任何地方的代码都能直接读写这些数据或调用这些函数；

protected：对外部代码完全封闭，但对继承链上的子类完全开放； 

private：权限最严格，只有当前类内部能访问，子类也休想直接碰；

| 访问来源 | public 成员 | protected 成员 | private 成员 |
| -- | --| -- | -- |
|当前类的成员函数 | ✅ 可以访问 | ✅ 可以访问 | ✅可以访问| 
|当前类的友元（friend）| ✅ 可以访问 | ✅ 可以访问 | ✅ 可以访问 |
| 子类（派生类）的成员函数| ✅ 可以访问 | ✅ 可以访问 | ❌ 无法访问 |
|类外部的普通函数 / 对象直接调用 |✅ 可以访问 | ❌  无法访问 | ❌ 无法访问 |


友元 friend ：权限在类本身，而不在外部

> 为了防止特权被滥用，C++ 对友元做出了严格的法律限制：\
> 1.友元关系是单向的：A 是 B 的友元，代表 A 可以看 B 的隐私；但不代表 B 可以看 A 的隐私。
> 2.友元关系是不能传递的：A 是 B 的友元，B 是 C 的友元，不代表 A 是 C 的友元（“ 朋友的朋友，不一定是 的朋友”）。
> 3.友元关系是不能继承的：父类的友元，到了子类那里自动失效；同样，子类的友元也无权访问父类（“父亲的挚友，不一定是儿子的挚友”）。

虽然友元破坏了面向对象的“封装性”，但在以下 3 个经典场景中，使用友元不仅不是坏习惯，反而是最佳或唯一的解决方案：

**1.运算符重载（最常见的场景）**：需要重载某些双目运算符（如输入输出流 << 和 >>）时，左操作数通常是系统的类（如 std::ostream），无法修改系统类的成员函数。此时必须将重载函数定义在全局，并将其设为当前类的友元，以便它能直接打印或读取 private 数据

```cpp
#include <iostream>

class Point {
private:
    int x, y;
public:
    Point(int x, int y) : x(x), y(y) {}

    // 声明全局函数为友元，使其能访问 x 和 y
    friend std::ostream& operator<<(std::ostream& os, const Point& p);
};

// 全局运算符重载函数
std::ostream& operator<<(std::ostream& os, const Point& p) {
    os << "(" << p.x << ", " << p.y << ")"; // ✅ 成功访问 private 成员
    return os;
}
```

**2. 两个或多个类高度紧密耦合（数据交互频繁）**：当两个类在业务逻辑上是“一体两面”的关系，需要高频交换私有数据时。如果通过 public 的 get()/set() 函数互传，会产生大量的函数调用开销（影响性能），且写起来极其繁琐。

```cpp
class Node {
private:
    int data;
    Node* next;

    // 将 Graph 设为友元类，Graph 可以直接操作 Node 的指针链
    friend class Graph; 
};

class Graph {
public:
    void insert(Node* n) {
        // Graph 可以直接修改 Node 的私有 next 指针
        n->next = nullptr; 
    }
};
```

**3.优化性能与设计简化**：优化性能与设计简化在某些底层开发或游戏引擎中，频繁通过虚函数或公共接口去获取私有变量会带来额外的开销。使用友元可以直接进行内存级别的读写，在保证局部代码整洁的同时，榨干运行性能。


## 3.虚函数
虚函数允许派生类取代基类所提供的实现。编译器确保当对象为派生类时，取代者（即派生类的实现）总是被调用，即使对象是使用基类指针访问而不是派生类的指针。这样就允许基类的算法被派生类取代，即使用户不知道派生类的细节。

派生类可以完全地取代基类成员函数（覆盖(override)），也可以部分地取代基类成员函数（增大(augment)）。如果愿意的话，后者由派生类成员函数调用基类成员函数来完成。


> 编译器底层是通过 虚函数表（vtable） 和 虚表指针（vptr） 来实现的。\
> vptr：当一个类包含虚函数时，编译器会在该类的对象内存空间中，秘密插入一个指针（vptr）。
> vtable：每个类都有一个虚函数表（vtable），里面记录着各个虚函数的真正地址。当派生类覆盖（override）了基类函数时，派生类虚表里的那个函数地址就被替换成了派生类自己的函数地址。

**1.完全地取代基类成员函数（Override / 覆盖）**

基类只定义一个规范或默认行为，子类彻底推翻重写，完全用自己的逻辑代替父类
```cpp
class Hero {
public:
    virtual void useUltimate() {
        // 基类的默认普通攻击
        std::cout << "释放普通大招" << std::endl;
    }
};

class Mage : public Hero {
public:
    // 完全取代：法师的大招完全不同，不需要父类的逻辑
    void useUltimate() override {
        std::cout << "召唤全屏暴风雪！" << std::endl;
    }
};
```

**2.部分地取代基类成员函数（Augment / 增大）**

子类既不想丢弃父类已经写好的成熟逻辑，又想在这个基础上增加自己特有的新步骤。

```cpp
class Window {
public:
    virtual void draw() {
        std::cout << "绘制窗口的基础边框" << std::endl;
    }
};

class ShadowWindow : public Window {
public:
    // 部分取代（增大）：先让父类画好边框，自己再补上阴影
    void draw() override {
        Window::draw(); // 显式调用基类函数，复用核心逻辑
        std::cout << "在其下方绘制立体阴影效果" << std::endl; // 增大（扩展）
    }
};
```


### 3.1 开闭原则 (OCP)
 “开闭原则”（Open-Closed Principle） ：
 - 对扩展开放：新同事可以通过编写新的派生类，写出完全不同或者功能更强大的算法（完全取代或部分取代）。
 - 对修改封闭：原本已经写好的老系统（使用基类指针管理对象的那些核心代码），不需要做任何修改，甚至不需要重新编译，就能直接跑起新同事写的新算法。


### 3.2 动态绑定和静态类型
在 C++ 中，动态绑定（Dynamic Binding）和静态类型（Static Type）并不是相互对立的，而是同时并存于同一个指针或引用上的。

对于一个用于实现多态的基类指针或引用：
1. 它的静态类型在编译期就固定了（决定了能调用哪些接口）。
2. 它的动态类型在运行期根据实际对象决定（通过虚函数表，决定了具体执行哪个版本的代码）。

```cpp
class Base {
public:
    virtual void show() { std::cout << "Base Show" << std::endl; }
    void baseOnly() { std::cout << "Base Only" << std::endl; }
};

class Derived : public Base {
public:
    void show() override { std::cout << "Derived Show" << std::endl; }
    void derivedOnly() { std::cout << "Derived Only" << std::endl; }
};

int main() {
    // 关键点：p 的静态类型是 Base*，但动态绑定指向了 Derived 对象
    Base* p = new Derived(); 

    // 1. 动态绑定的体现
    p->show();        // 输出 "Derived Show"
                      // 运行时通过虚函数表（Vtable）动态绑定到了子类实现

    // 2. 静态类型的限制
    p->baseOnly();    // 成功调用。因为 p 的静态类型是 Base*，拥有该接口
    // p->derivedOnly(); // 编译报错！因为 p 的静态类型是 Base*，编译器找不到该接口

    delete p;
}
```

底层协作：

1. 静态类型（Static Type）—— 编译期的安全哨兵 
    - 定义：变量在代码中被声明的类型。例如上面的 Base* p，p 的静态类型就是 Base*。
    - 作用：编译器在编译代码时，只看静态类型。它会检查 Base 类里有没有 show() 或 baseOnly() 这两个函数。如果没有，直接报错。这保证了代码的类型安全。

2. 动态绑定（Dynamic Binding）—— 运行期的执行向导
    - 定义：指针实际指向的对象的类型。上面的 p 实际指向 Derived 对象。
    - 作用：当编译器发现 调用的 show() 是一个 virtual（虚函数）时，它会生成一段特殊的机器码，在程序运行时去查找该对象头部的虚表指针（vptr），从而调用到 Derived::show()。


### 3.3 CRTP（编译期伪动态绑定）
希望既有继承和多态的接口复用感，又完全不想付出运行时虚函数表的性能代价，C++ 还有一种非常著名的方法叫做 **CRTP（奇异递归模板模式）**。

它利用模板，在编译期（静态）就完成了绑定：

```cpp
template <typename Derived>
class Base {
public:
    void interface() {
        // 静态向上转型，在编译期直接绑定到子类的方法
        static_cast<Derived*>(this)->implementation();
    }
};

class Derived : public Base<Derived> {
public:
    void implementation() { std::cout << "Derived Class" << std::endl; }
};

```

这种方式不需要 virtual 关键字，在编译后直接内联，没有任何运行时开销，被称为**静态多态**。


### 3.4 虚成员函数 vs 非虚成员函数
| 特性 | 非虚成员函数 (Non-virtual) | 虚成员函数 (Virtual) |
| :--- | :--- | :--- |
| **绑定时间** | **编译期**（静态绑定 / Early Binding） | **运行期**（动态绑定 / Late Binding） |
| **地址确定依据** | 取决于指针或引用的**静态类型**（声明的类型） | 取决于指针或引用指向的**实际对象类型** |
| **底层寻址机制** | 编译器直接硬编码函数地址（普通直接跳转） | 通过**虚函数表 (vtable)** 进行间接寻址 |
| **调用开销** | **极低**（与普通 C 函数相同，支持内联优化） | **略高**（存在两次指针解引用查表的开销） |

虚函数的动态查表
对于虚函数，编译器在编译时无法确定 `p` 实际指向哪个子类，因此会生成一段“查表机器码”：
```cpp
p->virtualFunc();
// 编译器的底层寻址逻辑：
// 1. 顺着 p 找到对象内存空间的头部
// 2. 取出头部隐藏的虚表指针（vptr）
// 3. 访问虚函数表（vtable），根据固定偏移量取出目标函数地址并执行
// *(p->_vptr[0])(p); 
```

**强调：基类的析构函数必须是虚函数（除非该类不打算作为多态基类使用）。**

> std::string 和 std::vector：它们的析构函数都不是虚函数。因为它们被设计成“高效率的独立工具类”，为了追求绝对的轻量化，它们没有引入 vptr。因此，在 C++ 中绝对不要去继承 std::string 或 std::vector，否则一旦发生向上转型和 delete，delete 删除基类就会有问题。

> 在现代 C++（C++11及以后）中，如果设计了一个类且不打算写虚析构函数，最安全的做法是加上 final 关键字，从根本上禁止他人继承它，防止误用：class MyClass final { ... }; // 安全，别人无法继承

### 3.5 Derived* —> Base* 可以很好地工作; 为什么 Derived** —> Base** 不行？
Derived* -> Base* 是绝对安全的；但如果允许 Derived** -> Base** 隐式转换，就会在不知不觉中彻底破坏 C++ 的类型安全系统，导致非法的内存写入和程序崩溃


```cpp

class Base { public: virtual void speak() {} };
class Derived : public Base {};
class AnotherDerived : public Base { 
public: 
    void uniqueMethod() { /* 特有方法 */ } 
};

int main() {
    Derived* d = new Derived();
    Derived** pd = &d; // pd 指向一个只能存放 Derived* 的变量

    // 假设下面这行代码能编译通过（实际上编译器会报错）：
    Base** pb = pd; 

    // 此时 pb 和 pd 指向同一个内存地址
    // 灾难发生：因为 pb 是 Base**，我们可以通过它把一个 AnotherDerived* 塞进去！
    *pb = new AnotherDerived(); 

    // 恐怖的后果：对于 pd 而言，它认为自己指向的依然是 Derived*
    // 但实际上，那块内存现在被换成了 AnotherDerived* ！
    pd[0]->speak(); // 也许还能跑，因为都是 Base 的虚函数
    
    // 彻底崩溃：调用了 Derived 根本没有、而 AnotherDerived 特有的方法，引发内存越界
    // 或者反过来，把 AnotherDerived 当作 Derived 来用，访问了不存在的成员变量
}
```

关键矛盾点
1. d 本身是一个类型为 Derived* 的变量，它只能且必须保存 Derived 对象的地址。
2. 如果允许 Base** pb = &d;，就拥有了一个可以写入 Base* 类型指针的通道。
3. 此时，可以通过 *pb 写入任何 Base 的子类（比如 AnotherDerived*）。
4. 这直接导致原本属于 Derived* d 的内存空间被悄悄掉包成了 AnotherDerived*，彻底绕过了 C++ 的类型检查！

一级指针为什么就可以？

因为一级指针的隐式转换会产生一个“右值副本（临时变量）”：
 - 当你执行 Base* b = d; 时，编译器并没有改变 d 本身，而是复制了 d 里面的地址，创建了一个全新的 Base* 类型的变量 b。
 - 你修改 b 的指向（如 b = new AnotherDerived();），只会改变 b 自己，绝对不会影响到原变量 d 的类型和内容。所以它是绝对安全的。


## 4. 抽象基类

抽象基类（Abstract Base Class, ABC）是不能被实例化的类，专门用作其他类的基类。它通过提供一个通用接口，来强制子类必须实现特定的行为。

纯虚函数：在虚函数声明末尾加上 = 0。类中只要有一个纯虚函数，该类就自动成为抽象基类。为了防止内存泄漏，抽象基类必须声明虚析构函数。

无法实例化：不能创建抽象基类的对象。

接口定义：定义子类必须遵循的契约。若子类没有实现所有的纯虚函数，则该子类也会变成抽象基类。

指针与引用：可以创建抽象基类的指针或引用，用于实现多态。

> 基类的非虚函数调用虚函数可以吗？ \
> 当基类的非虚函数调用虚函数时，依然会触发动态绑定（多态）。程序在运行时会根据实际指向的子类对象，去虚函数表中找到并执行子类重写后的虚函数。这种机制在面向对象设计中被称为 模板方法模式（Template Method Pattern）。

```cpp
#include <iostream>

class DrinkTemplate {
public:
    // 非虚函数：定义了固定的算法骨架（流程）
    void makeDrink() {
        boilWater();       // 固定的步骤
        brew();            // 变化的步骤：调用虚函数
        pourInCup();       // 固定的步骤
    }

    virtual ~DrinkTemplate() {}

private:
    void boilWater() { std::cout << "1. 将水烧开\n"; }
    void pourInCup() { std::cout << "3. 倒入杯中\n"; }

protected:
    // 虚函数：留给子类去具体实现
    virtual void brew() = 0; 
};

class Coffee : public DrinkTemplate {
protected:
    // 子类重写虚函数
    void brew() override { std::cout << "2. 冲泡咖啡粉\n"; }
};

class Tea : public DrinkTemplate {
protected:
    // 子类重写虚函数
    void brew() override { std::cout << "2. 浸泡茶叶\n"; }
};

int main() {
    DrinkTemplate* drink = new Coffee();
    // 调用基类的非虚函数
    drink->makeDrink(); 
    
    delete drink;
    return 0;
}
```

当执行 drink->makeDrink() 时，编译器会把当前对象的地址作为 this 指针传给 makeDrink()。此时 this 指针的实际类型是 Coffee*（尽管它被隐式转型为了 DrinkTemplate*）

在 makeDrink() 内部调用 brew() 时，等同于执行 this->brew()。因为 brew() 是虚函数，编译器生成的代码会去 this 隐式指向的对象头部寻找虚表指针（vptr），从而精准找到 Coffee 类的虚函数表并执行

> NVI 模式（Non-Virtual Interface，非虚接口模式） \
> 做法：将提供给外部调用的接口全部定义为非虚函数（如 makeDrink），而将具体允许子类定制的逻辑定义为 protected 或 private 的虚函数（如 brew）。\
> 好处：基类可以牢牢控制执行流程和权限（例如在调用 brew 前后做日志记录、参数检查、加锁解锁等），子类只需要专注于核心业务逻辑的实现，无法破坏整体框架。


## 5.虚函数表和虚表指针的区别

** 类有一张表，对象有一个指针**

**类（Class）级别**：只要类里有虚函数，这个类就会拥有一张虚函数表。这张表保存在内存的常量区（只读数据段/.rodata），在程序运行期间只有一份，被该类的所有对象共享。

**对象（Object）级别**：每个该类的对象并不会包含整张表，而是只在内存结构头部多出一个虚表指针（vptr），指向所属类的虚函数表。


如果基类已经有了虚函数表，那么继承它的派生类：
- 也会拥有属于自己独立的一张虚函数表。
- 即使派生类中没有新增任何新的虚函数，也没有重写（override）任何父类的虚函数，它依然有一张独立的表。只不过这张表里的函数地址，和父类虚函数表里的地址完全一样。
- 一旦子类重写了某个虚函数，子类虚表里对应的位置就会被替换为子类自己的函数地址。

> 在现代编译器的激进优化（如全程序优化/链接时优化 WPO/LTO）下，可能会出现有虚函数却没生成虚表的情况：\
> **1.无处引用的类**：如果一个类定义了虚函数，但整个工程中没有任何地方实例化过这个类的对象，链接器在代码裁剪（Dead code elimination）时，可能会直接把这个类以及它的虚表全部丢弃，不编入最终的二进制文件中；\
> **2.虚函数被完全内联**：如果编译器在静态分析时，发现虽然写了虚函数，但程序中所有调用它的地方全都是显式确定类型的（例如 Base b; b.func(); 而不是通过指针多态调用），编译器可能会直接把虚函数内联优化。此时如果不满足其他条件，虚表也可能被隐式省去。

### 5.1 虚表结构
可以把虚函数表理解为一个隐藏的 const void* vtbl[] 数组

第一项/前置项：通常存放的是 RTTI（运行时类型信息，用于 dynamic_cast 和 typeid 识别）。

后续项：依次存放着各个虚函数的函数指针；

    对象内存布局 [ vptr ] ───> 虚函数表 (vtbl)
            [ member1 ]          [ 0 ] ──> RTTI 信息 (type_info)
            [ member2 ]          [ 1 ] ──> Base::func1() 的地址
                                 [ 2 ] ──> Base::func2() 的地址

总结来说，在正常的程序逻辑中，只要定义了虚函数，这个类就必然对应一张虚函数表。

```cpp
class Animal {
public:
    int age = 4;
    
    virtual void speak() { } // 虚函数 1
    virtual void eat() { }   // 虚函数 2
};

class Cat : public Animal {
public:
    int score = 100;

    void speak() override { } // 重写基类虚函数 1
    virtual void meow() { }   // 子类新增的虚函数 3
};

```

【 myCat 对象的内存布局 】              【 Cat 类的虚函数表 (vtbl) 】
(位于栈区或堆区，每个对象独有一份)          (位于只读数据段 .rodata，全类共用一份)
┌─────────────────────────┐          ┌───────────────────────────────────┐
│ 0x00: vptr (虚表指针)   ├─────────>│ 0x00: RTTI 指针 (指向Cat的类型信息)│
├─────────────────────────┤          ├───────────────────────────────────┘
│ 0x08: Animal::age (4)   │          │ 0x08: &Cat::speak() [已重写，指向子类]
├─────────────────────────┤          ├───────────────────────────────────
│ 0x0C: Cat::score (100)  │          │ 0x10: &Animal::eat() [未重写，指向父类]
└─────────────────────────┘          ├───────────────────────────────────
                                     │ 0x18: &Cat::meow()  [子类新增的虚函数]
                                     └───────────────────────────────────

**对象的开头是 vpt**r：无论定义了多少个虚函数，myCat 对象内部只多占用 8 个字节。这 8 个字节是一个指针（vptr），它里面写着一个内存地址（比如 0x4000F0），这个地址就是 Cat 类的虚表开头。

**虚表的第一项**：RTTI（运行时类型信息）虚表并不是一上来就放函数指针的。它的第一项（索引 0）通常存放着一个指向 type_info 结构的指针。C++ 的 dynamic_cast 和 typeid 关键字，就是靠这个指针来判断当前对象到底是什么类型的。

**虚表的后续项**：
- 函数指针数组&Cat::speak()：因为 Cat 重写了 speak，所以虚表里原本属于父类的地址，被覆盖（Override）成了子类函数的地址。
- &Animal::eat()：因为 Cat 没有重写 eat，所以虚表里依然老老实实地记录着父类 Animal::eat 的函数入口地址。
- &Cat::meow()：这是子类自己新加的虚函数，它会被追加到虚表的末尾。

> 运行时它是怎么多态调用的
> ```cpp 
> Animal* ptr = new Cat();
> ptr->speak(); 
> ```
> 第一步：找到对象的虚表指针 -> void** vtbl = *(void***)ptr;\
> 第二步：从虚表中取出对应的函数地址（speak 在虚表里是扣除 RTTI 后的第 1 个函数，即索引 1） -> void (*func_ptr)() = (void(*)())vtbl[1];\
> 第三步：通过函数指针完成调用 -> func_ptr() \
> 因为 vtbl[1] 里面填的是 &Cat::speak()，所以虽然 ptr 是 Animal* 类型，执行的依然是猫叫。

## 6 多重继承
多重继承（Multiple Inheritance）是指一个派生类同时继承两个或两个以上的基类。

```cpp
class BaseA {
public:
    int a = 1;
    virtual void funcA() {}
    virtual void share() {} // 两个父类同名的虚函数
};

class BaseB {
public:
    int b = 2;
    virtual void funcB() {}
    virtual void share() {} // 两个父类同名的虚函数
};

// 多重继承
class Derived : public BaseA, public BaseB {
public:
    int d = 3;
    void share() override {}  // 同时重写两者的 share()
    virtual void funcD() {}   // 子类新增的虚函数
};
```

在 64 位系统下实例化一个 Derived obj; 时，这个对象内部会拥有 2 个虚表指针（vptr），并且编译器会为它生成 2 张虚函数表（vtbl）：

    【 obj 对象的内存布局 】                        【 Derived 类的两张虚函数表 】
    ┌───────────────────────────┐                ┌───────────────────────────────────┐
    │ 0x00: vptr_A (指向虚表A)   ├───────────────>│ 0x00: RTTI (Derived 类型信息)     │
    ├───────────────────────────┤                ├───────────────────────────────────┤
    │ 0x08: BaseA::a (1)        │                │ 0x08: &BaseA::funcA()             │
    ├───────────────────────────┤                ├───────────────────────────────────┤
    │ 0x10: vptr_B (指向虚表B)   ├───┐            │ 0x10: &Derived::share() [重写]    │
    ├───────────────────────────┤   │            ├───────────────────────────────────┤
    │ 0x18: BaseB::b (2)        │   │            │ 0x18: &Derived::funcD() [追加在此] │
    ├───────────────────────────┤   │            └───────────────────────────────────┘
    │ 0x20: Derived::d (3)      │   │            ┌───────────────────────────────────┐
    └───────────────────────────┘   └───────────>│ 0x00: RTTI (Derived 类型信息)     │
                                                ├───────────────────────────────────┤
                                                │ 0x08: &BaseB::funcB()             │
                                                ├───────────────────────────────────┤
                                                │ 0x10: &thunk to Derived::share()  │
                                                └───────────────────────────────────┘


**主基类与次基类**：在继承列表中写在前面的类（BaseA）被称为主基类。子类新增的虚函数（如 funcD()）会被追加到主基类的虚函数表末尾。

**对象里有几个 vptr？**：有几个包含虚函数的父类，子类对象里就有几个 vptr。在本例中，obj 内部有 vptr_A 和 vptr_B。

**this 指针修正（Thunk 机制）**：
```cpp
BaseB* ptr = &obj; // 此时 ptr 指向对象的 0x10 位置（BaseB的起始位置）
ptr->share();       // 调用 share()
```

由于 share() 被子类重写了，它需要执行 Derived::share()。但 Derived::share() 内部可能会访问 BaseA::a 或 Derived::d，它默认传入的 this 指针应该指向对象的开头（0x00）

为了让指针“对齐”，第二张虚表里存的不是直接的函数地址，而是一段被称为 Thunk 的汇编小代码。它的作用是：把当前 this 指针减去偏移量（这里是减去 16 字节），将其调整回 0x00，再去真正调用 Derived::share()。

## 7 虚继承
虚继承（Virtual Inheritance）是 C++ 专门用来解决多重继承中“菱形继承（Diamond Inheritance）”带来的一系列问题的终极武器。

**确保在多重继承的闭环中，最顶层的公共基类在子类的内存中只存在一份拷贝。**

菱形继承示例：

```cpp
class Animal { public: int age = 10; };
class Donkey : public Animal { }; // 驴
class Horse  : public Animal { }; // 马

class Mule : public Donkey, public Horse { }; // 骡
```

此时的两个致命问题：
- **数据二义性**：如果你写 Mule m; m.age = 5;，编译器直接报错 request for member 'age' is ambiguous。因为 m 内部同时有一份“驴的 age”和一份“马的 age”，编译器不知道你想改哪一个。
- **内存浪费**：公共基类 Animal 的成员被无意义地复制了两份，浪费了内存。

解决办法非常简单，只需在中间一层的继承关系上加上 virtual 关键字：

```cpp
class Animal { public: int age = 10; };

// 注意这里加了 virtual
class Donkey : virtual public Animal { }; 
class Horse  : virtual public Animal { }; 

// 此时 Mule 继承下来，内部就只有一份 Animal 了
class Mule : public Donkey, public Horse { }; 
```

虚继承下的内存：
- 虚继承能够让子类共享同一个祖先，是因为编译器彻底重构了对象的物理内存布局。在普通继承下，子类内存中直接“嵌套”了父类的全部成员。
- 但在虚继承下，编译器把公共基类（Animal）的数据扔到了对象内存的最末尾，并在中间层（Donkey/Horse）中插入了一个隐藏的**虚基类指针（vbptr，Virtual Base Pointer）**。

    【 mule 对象的物理内存 】                       【 虚基类表 (vbtbl) 】
    ┌──────────────────────────┐                  
    │ 0x00: vbptr_Donkey       ├───────────────> [ 记录了偏移量：距离末尾 Animal 的距离 ]
    ├──────────────────────────┤                  
    │ 0x08: vbptr_Horse        ├───────────────> [ 记录了偏移量：距离末尾 Animal 的距离 ]
    ├───────────────────────────┤
    │ 0x10: 唯一的 Animal::age  │ <── 所有人共享
    └───────────────────────────┘

> 无论代码中是通过 Donkey 指针还是 Horse 指针去访问 age，程序在运行时都会先读取各自对应的 vbptr，从虚基类表（vbtbl）中查出 Animal 成员的相对偏移量，然后再通过这个偏移量精准定位到末尾那份唯一的 Animal::age。

**一个子类对象中可以既有虚指针（vptr，指向虚函数表），又有虚基类指针（vbptr，指向虚基类表）**。
- vptr（虚指针）：为了实现动态多态（因为类里有 virtual 虚函数）。
- vbptr（虚基类指针）：为了实现虚继承（因为用了 virtual public 继承，需要消除菱形继承的二义性）。

> **编译器的现代优化：二合一（vptr 兼职 vbptr）**：在 Linux 下使用现代的 GCC 或 Clang 编译器，当尝试去打印上述对象的 sizeof(Donkey) 时，可能会发现一个有趣的现象：有些编译器下，vptr 和 vbptr 被合并了！ \
> 1.它把虚基类的偏移量信息，直接以负数索引的形式，逆向塞进了虚函数表（vtbl）的头部（也就是 RTTI 指针的更上方）。 \
> 2.此时，对象内部只需要保留一个 vptr，它既能用来查虚函数实现多态，又能用来查虚基类偏移量实现虚继承。这种优化进一步压榨了内存，完美践行了“零开销原则”。
